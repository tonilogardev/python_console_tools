# Contribuir en python_console_tools\n\nSigue estas pautas para mantener calidad y ritmo. Para detalle completo ver [docs/engineering-guidelines.md](docs/engineering-guidelines.md).\n\n## Flujo de ramas\n- Rama base estable: `main`.\n- Desarrollo continuo: `codex/main_dev_pro`.\n- Trabajo de interfaz actual: `codex/main_dev_pro_interface`.\n- Nuevas ramas: `codex/<area>-<breve>` (ej. `codex/feat-auth`).\n\n## Pasos antes del PR\n1. Instala deps: `pip install -e .[dev]` (en venv) y `pre-commit install`.\n2. Ejecuta `make format lint type test` (o los comandos equivalentes). Cobertura objetivo ≥85%.\n3. Corre `bandit -r src` si tocas IO, subprocess, seguridad.\n4. No subas secretos ni `.env`.\n\n## Commits y PR\n- Commits estilo conventional: `feat: ...`, `fix: ...`, `chore: ...`, `docs: ...`, `test: ...`.\n- PR debe explicar qué cambia, por qué y riesgos; enlaza issue si aplica.\n- Incluye checklist de checks ejecutados.\n\n## Estilo de código\n- Typing obligatorio; `mypy` debe pasar.\n- Lint/format con `ruff` (ver `pyproject.toml`).\n- Usa logging en lugar de `print`; carga config desde `configs/logging.dev.yaml`.\n- Valida entradas en la CLI (Typer) y proporciona `--help` claro.\n\n## Tests\n- Añade tests para funciones públicas nuevas o cambiadas.\n- Usa `pytest` + fixtures; evita depender de red/tiempo real.\n\n## Cómo pedir merge\n- Abre PR desde tu rama hacia `codex/main_dev_pro` o `main` según el caso.\n- Espera revisión o auto-aprúebalo si eres responsable y los checks pasan.\n*** End Patch
